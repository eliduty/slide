# AI Coding 实践分享 - 演讲说辞

## Part 1: 开场（3-5 分钟）

### 1. 封面
**说辞**：
"大家好！今天我要分享的主题是 'AI Coding 实践分享'，副标题是 '从对话到 Agent，重新定义开发流程'。这是一场关于 AI 如何改变我们开发方式的分享。"

### 2. AI 编程的现状与痛点
**说辞**：
"在开始之前，我想先问大家几个问题：

- 你们在使用 AI 编程工具时，有没有遇到过 AI 生成的代码不符合项目规范的情况？
- 有没有遇到过 AI 不理解你的项目架构，生成的代码无法集成的情况？
- 有没有遇到过 AI 写的代码难以维护的情况？

这些问题的根源其实很简单：传统的 AI 工具缺乏上下文理解，没有结构化的工作流程，也缺乏规范约束。"

### 3. AI Coding 的现状与价值
**说辞**：
"AI Coding 已经从 '辅助工具' 进化为 '开发伙伴'。对我来说，Skill 的出现让我开始重新审视传统的软件开发流程。

今天的分享，我会带大家了解 AI Coding 的演进历程，介绍 Agent 能力模型，分享 Spec-Driven Development 方法论，展示真实的项目实战案例，最后给出一些行动建议。"

## Part 2: AI Coding 演进史（8-10 分钟）

### 4. 发展历程
**说辞**：
"让我们先回顾一下 AI Coding 的发展历程。从 2022 年到 2025 年，短短四年时间，AI 编程工具经历了四个阶段：Web对话 → 编辑器插件 → AI IDE → 终端Agent。"

### 5. 每个阶段的特点
**说辞**：
"2022 年是 Web 对话时代，我们只能在浏览器里和 AI 聊天，然后复制粘贴代码，上下文是断裂的。

2022-2023 年进入编辑器插件时代，GitHub Copilot 等工具出现，AI 可以在 IDE 里实时补全代码，但仍然受限于单文件上下文。

2024 年，AI IDE 时代到来，Cursor、Windsurf、Trae 等工具提供了一体化体验，AI 可以理解整个代码库。

2025 年，我们进入终端 Agent 时代，Claude Code 等工具可以深度理解项目，自主执行复杂任务。"

### 6. 主流 AI Coding 工具
**说辞**：
"现在市面上有很多 AI Coding 工具，主要分为两类：GUI 工具和命令行工具。

GUI 工具包括 Cursor、GitHub Copilot、Windsurf、Trae、Qoder、CodeBuddy 等。

命令行工具包括 Claude Code、OpenCode、Codex CLI 等。"

### 7. 行业趋势
**行业趋势**：
"行业数据显示，2024 年已有 65% 的开发者使用 AI 编程工具，预计 2025 年这个比例会达到 85% 以上。

就单纯实现一个功能的场景来说，使用 AI 编程工具后，代码编写速度提升 30-50%，代码质量提升 25%，项目交付时间缩短 20%。

更重要的是，行业正经历从 '模型崇拜' 向 '工程落地' 的深刻转型。单纯依靠增加模型参数规模已无法解决复杂业务逻辑中的幻觉与失控问题，上下文管理成为 AI 编码的核心挑战，规格驱动开发（SDD）正在成为企业级落地的关键范式。"

## Part 3: AI Agent 能力模型（10-12 分钟）

### 8. AI Agent 核心组件
**说辞**：
"接下来，我们来了解 AI Agent 的核心组件。一个完整的 Agent 系统包含六个部分：Agent、MCP、Skill、Memory、Rules、Commands。

简单来说，Agent 是智能体主体，负责决策和规划；MCP 是调度协议，连接所有模块；Skill 是能力模块，执行具体任务；Memory 是记忆状态，存储上下文信息；Rules 是规则约束，定义行为边界；Commands 是指令动作，具体执行操作。"

### 9. 以 TRAE 为例讲解
**说辞**：
"让我们以 TRAE 为例来理解 Agent 能力。TRAE 有两种主要模式：IDE 模式和 Solo 模式。

IDE 模式包含 Chat 和 Builder 两种交互方式：Chat 模式用于侧边对话，解答疑问；Builder 模式用于从 0 到 1 构建项目。

Solo 模式则是一个独立的 AI 工程师，能够自主执行复杂的开发任务，形成理解需求 → 自主规划 → 调用工具 → 完成任务的能力闭环。"

### 10. 关键概念解释
**说辞**：
"为了更好地理解这些概念，我给大家做个类比：

- Agent 就像一个数字员工，能自主完成任务
- Skill 就像专业技能，如编程、设计、测试
- Commands 就像动作指令，如读取文件、执行命令
- Memory 就像大脑记忆，存储经验和知识
- Rules 就像行为准则，确保安全和合规
- MCP 就像一个开放的通信协议，让 AI 代理能够安全地访问外部数据源"

### 11. Skill 的意义
**说辞**：
"Skill 的出现意义重大。它让 AI 编程从 '对话' 升级到 '结构化工作流'，从随意的代码生成到有组织的任务执行。

Skill 是可复用、可组合的能力单元，一个 Skill 可以在多个场景中使用，多个 Skill 可以组合完成复杂任务。

更重要的是，Skill 让我们重新审视开发流程，从 '人写代码' 转变为 '人定义规范，AI 执行'，开发者的角色从执行者转变为设计者。"

### 12. Agent 模式的优势与挑战
**说辞**：
"Agent 模式的优势很明显：
- 自主性：AI 能自主完成复杂任务
- 可预测性：基于规范执行，结果可控
- 可追溯性：完整的执行记录

但也面临一些挑战：
- 上下文管理：处理长上下文的效率
- 安全性：确保 AI 操作的安全
- 可控性：平衡自主性和人工控制"

## Part 4: Spec-Driven Development（8-10 分钟）

### 13. 什么是 SDD？
**说辞**：
"现在我们来了解 Spec-Driven Development，简称 SDD。简单来说，就是先定义 '是什么'，再定义 '怎么做'。

传统开发是：需求 → 手写代码 → 测试
SDD 是：需求 → 规范 → AI 生成 → 验证

SDD 关注的是需求定义和规范，而不是直接的代码实现。通过规范驱动，我们可以确保 AI 生成的代码符合预期，提高质量和可维护性。"

### 14. OpenSpec 讲解
**说辞**：
"OpenSpec 是由 Fission AI 开发的轻量级规范框架，专为 AI 编码时代设计。它的核心理念是：在写代码之前，先就'要构建什么'达成一致。

**它解决了什么问题呢？**
传统 AI 编程有个大问题：需求只在聊天记录里，结果不可预测，多个功能混在一起，团队难以协作。OpenSpec 通过独立的规范文件夹、明确的规范文档、每个变更独立的结构，解决了这些问题。

**与传统方式对比：**
传统方式是直接描述需求，AI 生成代码，结果不确定，需要反复修改。OpenSpec 方式是先创建规范文档，团队达成一致，然后 AI 按规范执行，结果可预测。

**市面上支持 SDD 的工具对比：**
- GitHub 的 Spec Kit 很彻底但太重，有刚性阶段门
- AWS 的 Kiro 强大但锁定在特定 IDE
- OpenSpec 更轻量，支持灵活迭代，支持 20+ 工具，自由选择

**OpenSpec 提供两种工作流模式：**

第一种是默认模式，也叫 Core Profile，适合小型功能和快速迭代。包含四个命令：propose 创建变更、explore 探索需求、apply 执行任务、archive 归档变更。

第二种是扩展模式，也叫 Expanded Profile，适合大型功能和团队协作。包含更多命令：new 创建脚手架、continue 逐个创建文档、ff 快速创建所有文档、verify 验证实现、sync 同步规范、bulk-archive 批量归档。

**重点介绍默认模式的流程：**
第一步是 explore，可选的，用于探索想法和调查问题。第二步是 propose，创建变更文件夹，自动生成 proposal.md、specs、design.md、tasks.md 四个文档。第三步是 apply，读取 tasks.md 按顺序实现任务。第四步是 archive，归档完成的变更。

**生成的文档结构包括：**
- proposal.md：变更提案，说明为什么要做、目标、范围、风险
- specs：规格说明，包含用户场景、功能需求、验收标准
- design.md：技术设计，包含架构、接口、数据模型
- tasks.md：实现任务，包含任务列表和完成标准"

### 15. Superpowers 讲解
**说辞**：
"Superpowers 是一个由 Jesse Vincent 开发的插件，为 AI 编程提供结构化工作流程。它于 2025 年 10 月发布，截至 2026 年 3 月已获得超过 127K Star。

Superpowers 的核心理念是：AI 先规划、再测试、后实现，结果可预测。它由 8 个技能组成，按完整工作流顺序排列：

1. **brainstorm（头脑风暴）**：通过苏格拉底式提问精炼需求
2. **use-git-worktrees（使用 Git Worktrees）**：创建隔离的工作空间
3. **write-plan（编写计划）**：制定详细的开发计划
4. **execute-plan（执行计划）**：按照计划执行任务
5. **subagent-driven-development（子代理驱动开发）**：每个任务独立上下文
6. **test-driven-development（测试驱动开发）**：先写测试，再写代码
7. **systematic-debugging（系统化调试）**：结构化的调试方法
8. **request-code-review（请求代码审查）**：自动请求代码审查

有了 Superpowers，AI 不再是直接写代码，而是先规划、再测试、后实现，结果更加可预测。"

### 16. 完整方案
**说辞**：
"我们的完整方案是：Claude Code + OpenSpec + Superpowers。

- Claude Code 提供深度代码理解和自主执行能力
- OpenSpec 提供结构化规范，确保可预测性
- Superpowers 提供自动化能力增强，保证质量

这三者配合工作：OpenSpec 生成规范文档，Superpowers 审查和优化规范，Claude Code 根据规范执行任务，Superpowers 验证执行结果。"

## Part 5: 项目实战流程（12-15 分钟）

### 17. 项目背景与数据概览
**说辞**：
"接下来，我要分享一个真实的项目案例：档案编研智能辅助选题系统。

这个系统面向综合档案馆的编研人员，覆盖省、市、县三级馆场景。从需求到落地，几乎全部由 AI 完成。

项目数据非常惊人：
- 开发周期：10 天
- 净增代码：25,546 行
- 工具调用：2,754 次
- 文件读取：738 次
- 代码编辑：550 次
- 终端命令：662 次
- 任务标记：208 次"

### 18. 开发时间线
**说辞**：
"项目开发分为四个阶段：

第一阶段是设计阶段，我们确认了产品方向，输出了 UI 设计稿和 PRD 需求描述。

第二阶段是项目搭建，用了 2 个工作日，20 条指令，完成了基础设施搭建和第一个核心页面。

第三阶段是功能开发，用了 4 个工作日，89 条指令，引入了 Spec Coding 工作流，完成了授权管理、数据分析看板等功能。

第四阶段是细节打磨与部署，用了 4 个工作日，108 条指令，进行了功能迭代、系统重构和部署排障。"

### 19. 核心工作流
**说辞**：
"项目的核心工作流是：需求 → 设计 → 开发 → 测试 → 部署。

开发阶段的详细流程是：
1. PRD功能模块
2. propose（生成提案、设计、规格、任务文档）
3. explore（探索代码库）
4. apply（执行，触发 superpowers）
5. archive（归档）"

### 20. 典型案例演示
**说辞**：
"我想分享两个典型案例：

第一个是 AI 驱动产品设计。我们没有产品经理和 UI 设计师，而是让 AI 扮演不同角色：
- AI 扮演产品经理：进行需求分析，设计用户路径，生成 PRD
- AI 扮演 UI 设计师：生成高保真 HTML 设计稿
- AI 生成研发可读的 PRD

第二个是 SDD 驱动前端功能研发。我们使用 OpenSpec 生成规范文档，Superpowers 自动审查规范，Claude Code 根据规范生成代码，通过测试驱动开发确保质量。"

### 21. 方法论融合
**说辞**：
"在项目中，我们融合了多种方法论：

- SDD：规范驱动，定义 '是什么'
- DDD：领域驱动，理解业务逻辑
- TDD：测试驱动，确保质量

我们还应用了金字塔测试模型：单元测试验证函数和组件，集成测试验证模块间交互，E2E 测试验证完整流程。
这一套流程是按照软件工程的开发流程进行设计的，去探索各种各样的工具，他确保了代码的质量和可维护性。在以前这一套流程落地非常困难，需要配置多个框架和流程的配合。并且绝大部分的人都做不好，然而在AI时代，我们可以让AI很顺利的完成。
同时，我们将整个流程集成到 CI/CD 中，实现了代码自动检测、测试自动运行和部署自动化。"

### 22. 经验总结与建议
**说辞**：
"通过这个项目，我们总结了一些经验：

做得好的地方：
- 规范驱动，减少了返工
- 工具链协同，提高了效率
- 数据驱动，量化了效果
- 方法论融合，保证了质量

踩过的坑：
- 初期规范不够详细，导致多次修改
- 工具调用过多，影响效率
- 复杂逻辑需要人工干预

给团队的建议：
- 从小项目开始尝试 SDD
- 建立团队规范模板
- 定期总结和优化工作流
- 培养 '规范先行' 的思维"

## Part 6: 总结（3-5 分钟）

### 23. Key Takeaways
**说辞**：
"最后，我想总结三个关键要点：

第一，AI Coding 已进化为 Agent 模式。从辅助工具到开发伙伴，AI 现在能够自主执行复杂任务，结果更加可预测。

第二，SDD 让人机协作更高效。通过先定义 '是什么'，再定义 '怎么做'，我们可以确保 AI 生成的代码符合预期，提高质量和可维护性。

第三，工作流变革 > 工具选择。工具只是手段，工作流才是核心。选择适合团队的工作流，比单纯追求最新工具更重要。"

### 24. 下一步行动建议
**说辞**：
"那么，如何开始尝试 AI Coding 呢？

首先，从一个小功能开始，选择合适的工具组合，建立简单的规范模板。

其次，推荐的学习路径是：学习 OpenSpec 基本使用，尝试 Superpowers 核心技能，实践 Spec-Driven Development。

最后，团队落地可以通过组织内部培训，建立规范库和最佳实践，定期分享和迭代。"
