## 文章一：Claude Code + OpenSpec 正在加速 AICoding 落地

> 来源：掘金 - 得物技术
> 链接：https://juejin.cn/post/7620583438190444578

### 核心观点

**AI 编码的真正瓶颈不是模型，是上下文管理。**

### 主要内容

#### 1. 生产力悖论

- 开发者使用 AI 时主观感知速度提升 20%，但实际完成任务时间增加 19%
- AI 采用率每增加 25%，交付稳定性反而下降 7.2%（DORA 2024 报告）
- GPT-4o 在 1K Token 时准确率 99.3%，扩展到 32K Token 时暴跌至 69.7%

#### 2. 上下文工程

上下文工程正在取代提示词工程，核心是"如何为模型筛选最精准的 Token 集合"。

关键原则：
- 存储与呈现分离
- 显式转换
- 默认作用域（每个子代理仅看到执行任务所需的最小上下文）

#### 3. Claude Code 核心特性

**代理循环（Agentic Loop）**：
```
Gather Context → Take Action → Verify Results
```

| 阶段 | 说明 |
|------|------|
| 收集上下文 | 文件搜索、Git 状态检查、读取 CLAUDE.md |
| 采取行动 | 跨文件编辑、终端工具操作 |
| 验证结果 | 自动运行测试、捕捉错误、调整方案 |

**终端原生优势**：
- 遵循 Unix 哲学
- 支持管道、脚本化、自动化集成
- 与 CI/CD 流程衔接

**MCP 整合**：
- 按需加载策略
- 工具搜索和代码执行模式
- 避免一次性加载大量 API 定义

**CLAUDE.md + Auto Memory**：
- CLAUDE.md：项目操作手册（编码标准、架构决策、测试指令）
- MEMORY.md：自动记忆构建命令、调试心得、用户偏好

#### 4. OpenSpec 核心概念

**规格驱动开发（SDD）**：在写任何代码之前，先锁定机器可读、人可评审的规格文档。

**工件体系**：
| 文件 | 说明 |
|------|------|
| proposal.md | 变更的初衷和范围 |
| specs/ | 逻辑规格，包含场景描述 |
| design.md | 技术设计方案 |
| tasks.md | 原子化任务清单 |

**生命周期管理**：
```
Proposal → Apply → Archive
```

- 解决上下文污染问题
- 归档后核心规格更新至主规格文件

#### 5. 实战流程

```
第一步：/opsx:propose → 生成骨架和规格
第二步：审阅 spec.md → 确认边界条件
第三步：/opsx:apply → 逐项实施并验证
第四步：/opsx:archive → 知识固化
```

#### 6. 角色转变

从"码农"到"规格定义者"：
- 代码变成编译后的中间产物
- 规格才是核心产品
- 开发者转向关注系统设计和逻辑严密性

## 文章二：OpenSpec 完全指南

> 来源：TinyAsh 博客
> 链接：https://www.tinyash.com/blog/openspec-ai/

### 核心定位

**OpenSpec 是一个轻量级规范框架，专为 AI 编码时代设计。**

> 在写代码之前，先就"要构建什么"达成一致。

### 核心哲学

```
→ fluid not rigid          (灵活而非僵化)
→ iterative not waterfall  (迭代而非瀑布)
→ easy not complex         (简单而非复杂)
→ built for brownfield     (为现有项目设计)
→ scalable                 (从个人项目到企业级)
```

### 解决的问题

| 痛点 | 传统方式 | OpenSpec 方案 |
|------|----------|---------------|
| 需求只在聊天记录 | 需求淹没在聊天历史中 | 独立的规范文件夹，可追溯 |
| 结果不可预测 | 模糊的提示导致不确定输出 | 明确的规范带来可预测结果 |
| 缺乏组织 | 多个功能混在一起 | 每个变更独立的文件夹结构 |
| 难以协作 | 个人与 AI 对话，团队不知情 | 规范文档可共享和审查 |

### OPSX 工作流

**Core Profile（核心模式）**：
| 命令 | 说明 | 何时使用 |
|------|------|----------|
| /opsx:propose | 创建变更和规划文档 | 开始新功能 |
| /opsx:explore | 探索想法，调查问题 | 需求不明确时 |
| /opsx:apply | 实现任务 | 文档完成后 |
| /opsx:archive | 归档完成的变更 | 功能完成后 |

**Expanded Profile（扩展模式）**：
- /opsx:new - 创建变更脚手架
- /opsx:continue - 逐个创建文档
- /opsx:ff - 快速创建所有规划文档
- /opsx:verify - 验证实现符合文档
- /opsx:sync - 同步增量规范到主规范

### 文档模板结构

**proposal.md**：为什么要做、目标、范围、风险、回滚计划

**specs/spec.md**：
```markdown
### 场景：成功路径
- **Given** [前置条件]
- **When** [用户操作]
- **Then** [预期结果]
```

**design.md**：架构概述、组件设计、数据设计、API 设计、安全考虑

**tasks.md**：原子化任务清单，作为 AI 执行路径图

### 支持的工具

OpenSpec 支持 20+ AI 编码助手：Claude Code、Cursor、Windsurf、Codex 等

### 最佳实践

**推荐做法**：
- ✅ 使用项目配置注入上下文
- ✅ AI 生成初稿，人工审查完善
- ✅ 保持文档简洁，聚焦关键决策
- ✅ 使用 Given/When/Then 格式编写场景

**避免做法**：
- ❌ 过度详细，难以维护
- ❌ 完全依赖 AI，不人工审查
- ❌ 文档与代码脱节

## 文章三：Claude Code + OpenSpec + Superpowers 实战案例

> 来源：微信公众号
> 链接：https://mp.weixin.qq.com/s/mEH3FJXlZ25xJdqoefjvEw

### 项目背景

**档案编研智能辅助选题系统**，服务对象是综合档案馆的编研人员与业务主管领导。

### 核心能力

- 馆藏资源分析：识别具备编研条件的专题方向
- 社会需求感知：结合查阅记录、热点纪念节点发现选题
- 选题智能评估：四维度自动打分
- 同级馆选题查重：避免重复立项
- 立项审批流转：多级审批流程
- 编研任务跟踪：自动生成任务卡，跟踪进度

### 三工具分工

| 工具 | 职责 |
|------|------|
| OpenSpec | 自然语言需求 → 工程规范文档 |
| Superpowers | 自动审查规范文档，发现缺陷并修复 |
| Claude Code | 根据规范文档 → 可运行的代码 |

### 实战流程

#### 第一步：OpenSpec 生成规范

```bash
/opsx:propose 我想为综合档案馆的编研人员做一个智能辅助选题系统...
```

生成内容：
- proposal.md
- design.md
- tasks.md（9 个任务组，55 个任务）
- specs/（6 个模块规格）

#### 第二步：Superpowers 智能审查

```bash
/superpowers:brainstorm 请你认真阅读 openspec 生成的所有规范文档，
从档案业务专家、系统架构师、前端交互设计师三个视角找出问题
```

**发现的问题**：
- 🔴 高优先级（6 条）：如"审批流程缺少暂缓状态"、"用户角色遗漏科室负责人"
- 🟡 中优先级（4 条）：如"纪念节点数据来源未说明"
- 🟢 低优先级（3 条）

**修复后改进**：
- 新增"与在编项目重叠度检测"
- 新增三种馆藏数据适配方案
- 新增"暂缓"状态
- 扩充四级角色
- AI 报告约束为档案馆场景
- 新增冷启动方案

任务组扩充为 12 个，总任务达到 78 个。

#### 第三步：Claude Code 代码实现

```bash
/opsx:apply
```

自动生成：
- 后端项目结构（FastAPI + PostgreSQL + pgvector）
- 前端项目结构（React + TailwindCSS）
- 核心功能代码（评分引擎、审批状态机、AI 报告）

### 核心优势总结

1. **Superpowers 的业务审查价值突出**：在代码开始前发现业务需求遗漏
2. **极高的效率**：传统方式 1-2 周，AI 方式一天内完成核心部分
3. **极高的标准**：文档、代码均遵循企业级规范
4. **高度可复现**：相同指令得到相同结果

### 一句话总结

> AI 不再只是写代码的助手，而是能够贯穿需求、设计、开发全流程的核心能力。OpenSpec 让需求标准化，Superpowers 提前消灭业务盲区，Claude Code 让代码自动化。
