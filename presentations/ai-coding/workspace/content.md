### 2. AI 编程的现状与痛点
**痛点 1**：AI 生成的代码不符合项目规范
- 现象：AI 生成的代码风格、命名规范与项目不一致
- 影响：需要大量手动修改，降低开发效率

**痛点 2**：AI 不理解你的项目架构
- 现象：AI 生成的代码无法与现有架构集成
- 影响：代码难以维护，容易引入 bug

**痛点 3**：AI 写的代码难以维护
- 现象：生成的代码缺少注释、逻辑混乱
- 影响：后续维护成本高，团队协作困难

**问题根源**：
- 传统 AI 工具缺乏上下文理解
- 没有结构化的工作流程
- 缺乏规范约束

### 3. AI Coding 的现状与价值
**进化历程**：
- AI Coding 已从"辅助工具"进化为"开发伙伴"
- Skill 的出现是关键转折点，让 AI 能够执行结构化任务

**分享目标**：
- 分享 AI Coding 工作流变革的实践经验
- 介绍 Agent 模式和 SDD 方法论
- 展示真实项目案例和效果

**分享结构**：
- 演进史 → Agent 能力模型 → SDD 方法论 → 项目实战 → 总结

## Part 2: AI Coding 演进史（8-10 分钟）

### 4. 发展历程
**时间轴**：2022-2025
**阶段演进**：Web对话 → 编辑器插件 → AI IDE → 终端Agent

### 5. 每个阶段的特点
**Web对话时代（2022）**
- 代表工具：ChatGPT、Claude.ai
- 特点：上下文断裂，复制粘贴
- 工作流：描述需求 → AI 生成代码 → 复制到编辑器 → 手动调试

**编辑器插件时代（2022-2023）**
- 代表工具：GitHub Copilot、Codeium
- 特点：集成但受限
- 工作流：写注释/函数名 → Tab 补全 → 手动调整

**AI IDE 时代（2024）**
- 代表工具：Cursor、Windsurf、Trae
- 特点：一体化体验
- 工作流：自然语言描述需求 → AI 多文件修改 → 验证

**终端Agent时代（2025）**
- 代表工具：Claude Code、OpenCode
- 特点：深度理解，自主执行
- 工作流：自然语言指令 → AI 自主规划 → 执行 → 验证

### 6. 主流 AI Coding 工具对比
**GUI 工具**：
- **Cursor**：上下文工程，Composer 模式
- **GitHub Copilot**：实时补全，多语言支持
- **Windsurf**：Cascade 代理，Memories 系统
- **Trae**：国产模型，完全免费
- **Qoder**：专注前端开发

**命令行工具**：
- **Claude Code**：代理循环，自动验证
- **OpenCode**：开源替代，轻量级
- **Codex**：早期代码模型

**适用场景分析**：
- 个人开发：GitHub Copilot、Cursor
- 企业级项目：Trae、Claude Code
- 快速原型：Cursor、Windsurf

### 7. 行业趋势
**AI 编程工具采用率**：
- 2024年：65% 的开发者使用 AI 编程工具
- 2025年：预计达到 85%+

**开发效率提升数据**：
- 代码编写速度提升 30-50%
- 代码质量提升 25%
- 项目交付时间缩短 20%

**从"模型崇拜"到"工程落地"**：
- 行业正经历从单纯追求模型能力向工程化落地的转型
- 上下文管理成为 AI 编码的核心挑战
- 规格驱动开发（SDD）成为企业级落地的关键范式

## Part 3: AI Agent 能力模型（10-12 分钟）

### 8. AI Agent 核心组件
**架构图**：
```
┌───────────────────────────────────────────────────┐
│  Agent │ MCP │ Skill │ Memory │ Rules │ Commands  │
└───────────────────────────────────────────────────┘
```

**组件解释**：
- **Agent**：智能体主体，负责决策和规划
- **MCP**：调度协议，连接所有模块
- **Skill**：能力模块，执行具体任务
- **Memory**：记忆状态，存储上下文信息
- **Rules**：规则约束，定义行为边界
- **Commands**：指令动作，具体执行操作

### 9. 以 TRAE 为例讲解
**TRAE 的两种主要模式**：
- **IDE 模式**：包含 Chat + Builder
  - Chat：侧边对话，解答疑问
  - Builder：从 0 到 1 构建项目
- **Solo 模式**：独立 AI 工程师
  - 自主执行复杂开发任务
  - 能力闭环：理解需求 → 自主规划 → 调用工具 → 完成任务

**演示建议**：选择 Builder 模式，展示从 0 到 1 构建一个简单项目

### 10. 关键概念解释
**Agent**：智能体主体（决策、规划、控制）
- 类比：数字员工，能自主完成任务

**Skill**：能力模块（能做什么）
- 类比：专业技能，如编程、设计、测试

**Commands**：指令动作（具体调用什么）
- 类比：动作指令，如读取文件、执行命令

**Memory**：记忆状态（记住上下文、历史）
- 类比：大脑记忆，存储经验和知识

**Rules**：规则约束（权限、安全、边界）
- 类比：行为准则，确保安全和合规

**MCP**：模型上下文协议（开放标准，允许 AI 代理安全地访问外部数据源）
- 类比：神经系统，协调各部分工作

### 11. Skill 的意义
**从"对话"到"结构化工作流"**
- 传统 AI：纯对话式交互
- Skill：结构化任务执行

**可复用、可组合的能力单元**
- 一个 Skill 可以在多个场景中使用
- 多个 Skill 可以组合完成复杂任务

**重新审视开发流程的契机**
- 从"人写代码"到"人定义规范，AI 执行"
- 开发者角色转变：从执行者到设计者

### 12. Agent 模式的优势与挑战
**优势**：
- **自主性**：AI 能自主完成复杂任务
- **可预测性**：基于规范执行，结果可控
- **可追溯性**：完整的执行记录

**挑战**：
- **上下文管理**：处理长上下文的效率
- **安全性**：确保 AI 操作的安全
- **可控性**：平衡自主性和人工控制

## Part 4: Spec-Driven Development（8-10 分钟）

### 13. 什么是 SDD？
**一句话定义**：先定义"是什么"，再定义"怎么做"

**传统开发 vs SDD**
| 维度 | 传统开发 | SDD |
|------|----------|-----|
| 流程 | 需求 → 手写代码 → 测试 | 需求 → 规范 → AI 生成 → 验证 |
| 关注点 | 代码实现 | 需求定义和规范 |
| 质量控制 | 测试驱动 | 规范驱动 |
| 可维护性 | 依赖开发者经验 | 依赖规范文档 |

**为什么需要 SDD？**
- 解决 AI 编程的不可预测性
- 提供统一的质量标准
- 便于团队协作和知识沉淀

### 14. OpenSpec 讲解
**OpenSpec 是什么**：
- 由 Fission AI 开发的轻量级规范框架，专为 AI 编码时代设计
- 核心理念：**在写代码之前，先就"要构建什么"达成一致**
- 核心哲学：灵活而非僵化、迭代而非瀑布、简单而非复杂、为现有项目设计、可扩展

**解决的问题**：
| 问题 | 传统方式 | OpenSpec 方案 |
|------|----------|---------------|
| 需求只在聊天记录 | 需求淹没在聊天历史中 | 独立的规范文件夹，可追溯 |
| 结果不可预测 | 模糊的提示导致不确定的输出 | 明确的规范带来可预测的结果 |
| 缺乏组织 | 多个功能混在一起 | 每个变更独立的文件夹结构 |
| 难以协作 | 个人与 AI 对话，团队不知情 | 规范文档可共享和审查 |

**与传统方式对比**：
- **传统 AI 编程**：直接描述需求 → AI 生成代码 → 结果不确定 → 反复修改
- **OpenSpec 方式**：创建规范文档 → 达成一致 → AI 按规范执行 → 结果可预测

**市面上支持 SDD 的工具对比**：
| 工具 | 特点 | OpenSpec 优势 |
|------|------|---------------|
| Spec Kit (GitHub) | 彻底但重量级，刚性阶段门 | OpenSpec 更轻量，支持灵活迭代 |
| Kiro (AWS) | 强大但锁定在特定 IDE | OpenSpec 支持 20+ 工具，自由选择 |
| 无规范 | 快速但不可预测 | OpenSpec 带来可预测性，无繁文缛节 |

**两种工作流模式**：

1. **默认模式（Core Profile）** - 快速路径，适合小型功能和快速迭代
   - `/opsx:propose`：创建变更和规划文档
   - `/opsx:explore`：探索代码库和需求
   - `/opsx:apply`：执行任务，生成代码
   - `/opsx:archive`：归档完成的变更

2. **扩展模式（Expanded Profile）** - 完整流程，适合大型功能和团队协作
   - `/opsx:new`：创建变更脚手架
   - `/opsx:continue`：逐个创建文档
   - `/opsx:ff`：快速创建所有规划文档
   - `/opsx:verify`：验证实现是否符合文档
   - `/opsx:sync`：同步增量规范到主规范
   - `/opsx:bulk-archive`：批量归档多个变更

**默认模式流程详解**（重点介绍）：
- **步骤 1：/opsx:explore（可选）**：探索想法，调查问题，需求不明确时使用
- **步骤 2：/opsx:propose**：创建变更文件夹，自动生成 proposal.md、specs/、design.md、tasks.md
- **步骤 3：/opsx:apply**：读取 tasks.md，按顺序实现任务，生成代码
- **步骤 4：/opsx:archive**：归档完成的变更到 openspec/changes/archive/

**文档结构**：
- `proposal.md`：变更提案（为什么要做、目标、范围、风险）
- `specs/`：规格说明（用户场景、功能需求、验收标准）
- `design.md`：技术设计（架构、接口、数据模型）
- `tasks.md`：实现任务（任务列表、完成标准）

**演示建议**：展示如何使用 `/opsx:propose add-dark-mode` 命令生成规范文档

### 15. Superpowers 讲解
**Superpowers 是什么**：
- Superpowers 是一个由 Jesse Vincent 开发的插件，为 AI 编程提供结构化工作流程
- 发布时间：2025 年 10 月，Star 数：106,000+（截至 2026 年 3 月）
- 核心理念：AI 先规划、再测试、后实现，结果可预测

**核心技能**（8 个技能，按完整工作流顺序）：
1. **brainstorm（头脑风暴）**：通过苏格拉底式提问精炼需求
2. **use-git-worktrees（使用 Git Worktrees）**：创建隔离的工作空间
3. **write-plan（编写计划）**：制定详细的开发计划
4. **execute-plan（执行计划）**：按照计划执行任务
5. **subagent-driven-development（子代理驱动开发）**：每个任务独立上下文
6. **test-driven-development（测试驱动开发）**：先写测试，再写代码
7. **systematic-debugging（系统化调试）**：结构化的调试方法
8. **request-code-review（请求代码审查）**：自动请求代码审查

**演示建议**：选择 brainstorm 技能，展示如何通过提问精炼需求

**有 Superpowers vs 无 Superpowers**
| 场景 | 无 Superpowers | 有 Superpowers |
|------|----------------|----------------|
| 需求理解 | 直接写代码，不问细节 | 先提问，再设计 |
| 代码质量 | 没有测试，质量靠运气 | 先写测试，再写代码 |
| 执行过程 | 写到一半开始乱改 | 每个任务独立执行 |

### 16. 完整方案
**Claude Code + OpenSpec + Superpowers**
- **Claude Code**：深度代码理解，自主执行
- **OpenSpec**：结构化规范，可预测性
- **Superpowers**：自动化能力增强，质量保证

**三者如何配合工作**：
1. OpenSpec 生成规范文档
2. Superpowers 审查和优化规范
3. Claude Code 根据规范执行任务
4. Superpowers 验证执行结果

**架构图**：
```
┌───────────────────┐     ┌───────────────────┐     ┌───────────────────┐
│   OpenSpec        │────>│   Superpowers     │────>│   Claude Code     │
│   规范生成        │<────│   规范审查/优化   │<────│   代码执行/验证   │
└───────────────────┘     └───────────────────┘     └───────────────────┘
```

### 21. 方法论融合
**SDD + DDD + TDD 如何结合**
- **SDD**：规范驱动，定义"是什么"
- **DDD**：领域驱动，理解业务逻辑
- **TDD**：测试驱动，确保质量

**金字塔测试在 AI 编程中的应用**
- 单元测试：验证函数和组件
- 集成测试：验证模块间交互
- E2E 测试：验证完整流程

**CI/CD 集成**
- 代码自动检测
- 测试自动运行
- 部署自动化

### 22. 经验总结与建议
**哪些做得好**：
- 规范驱动，减少返工
- 工具链协同，提高效率
- 数据驱动，量化效果
- 方法论融合，保证质量

**哪些踩了坑**：
- 初期规范不够详细，导致多次修改
- 工具调用过多，影响效率
- 复杂逻辑需要人工干预

**给团队的建议**：
- 从小项目开始尝试 SDD
- 建立团队规范模板
- 定期总结和优化工作流
- 培养"规范先行"的思维
