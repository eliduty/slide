## 第1章：AI Coding工具的真实能力（15分钟）

### 1.1 能做什么

分类分级能力表（skill 项目实践）：

| 能力类型 | 强度 | 条件 | skill项目案例 |
|---------|------|------|--------------|
| 代码生成 | 强 | 需清晰规范 | Vue组件生成 |
| 架构决策 | 需人引导 | 明确约束 | Monorepo架构设计 |
| 跨模块重构 | 有风险 | 小步验证 | 多租户模块化 |
| 业务逻辑理解 | 依赖上下文 | 上下文质量 | DDD领域模型 |

### 1.2 哪些地方带来效率

skill 项目效率提升数据：

| 场景 | 传统耗时 | AI协作耗时 | 提升比例 |
|------|---------|-----------|---------|
| 接口契约设计 | 2天 | 半天 | 75% |
| UI 原型设计 | 1天 | 2小时 | 80% |
| 测试用例生成 | 3小时 | 30分钟 | 83% |

额外 review 成本环节：架构决策验证、跨模块影响分析

### 1.3 局限性是什么

- AI会"谄媚"：给出看起来合理但方向错误的建议
- 上下文有限：无法理解百万行代码的隐式依赖
- 缺乏业务直觉：不理解代码背后的业务意图
- 当AI做不好时，人应该怎么接管

## 第2章：基于AI Coding工具能力推导的工作流（20分钟）

### 2.1 标准工作流三阶段

基于 skill 项目的 OpenSpec 工作流：

**需求阶段（5W1H）**：
- 谁：PM + SDE
- 做什么：结构化需求描述
- 标准：业务边界明确、验收标准清晰
- 产出物：PRD文档
- 进入下一阶段条件：需求评审通过

**技术设计阶段（5W1H）**：
- 谁：SDE主导 + PM
- 做什么：技术方案、接口契约、测试策略
- 标准：技术选型确定、架构边界清晰
- 产出物：proposal.md + design.md + tasks.md
- 进入下一阶段条件：技术评审通过

**规范AI工作环境阶段（5W1H）**：
- 谁：SDE
- 做什么：开发、测试、联动调试、领域知识规范化
- 标准：CLAUDE.md配置、Skills加载
- 产出物：代码 + 测试 + 文档
- 进入下一阶段条件：lint通过、测试覆盖达标

### 2.2 人与AI的边界

**各阶段人/AI分工**：

| 阶段 | AI负责 | 人负责 |
|------|--------|--------|
| 需求 | 需求推演完善、结构化描述 | 业务目标定义、验收标准确定 |
| 设计 | 技术方案生成、接口契约设计 | 技术选型决策、架构边界定义 |
| 实现 | 根据规范生成代码、测试 | 验证输出质量、处理边界情况 |

## 第3章：对现有工作的影响+人的工作调整（10分钟）

### 3.1 对PM工作的影响

- 需求陈述的要求变化：结构化模板
- 一些设计工作可以由工具代劳：界面原型、数据流图
- 对界面设计的影响：AI可生成设计规范
- 知识文档的格式、内容范围和快速生成
- 需求变更如何实现：同步更新spec文件

### 3.2 对SDE/QA工作的影响

- 架构设计工作的前移与显性化
- SDE如何弥补测试技能的不足
- QA如何实现开发：基于tasks.md分配

### 3.3 对评审的影响

- 需求评审：重点验证AI推演结果的业务准确性
- Code Review：增加AI输出质量校验环节
- 测试评审：验证测试用例的业务意义

### 3.4 人的工作调整总结

- 向把关者和管理者转变：建立规则和标准
- 对上下文工程的理解：渐进式披露、分层摘要
- 职能全面化：自带研发全流程
- 与AI共生：既指导也采纳建议
- 问题分拆：架构能力
- 建立对AI输出的校准直觉

## 第4章：AI Coding工具作用于工作流（25分钟）

### 4.1 需求阶段

AI工具应用：
1. 基于少量需求的推演完善
2. 需求的结构化描述（markdown等）

skill项目案例：`docs/requirements/prd-admin-web.md`

### 4.2 设计阶段

AI工具应用：
1. 架构设计与项目结构治理
2. 技术选型（提供方案比较）
3. 界面交互原型设计（Pencil）
4. 界面设计规范与界面交互详细设计
5. 接口契约设计（OpenAPI）
6. 测试场景与测试方案制定
7. 运维与持续交付方案设计

skill项目案例：
- `openspec/changes/admin-role-authorization/design.md`
- `.claude/skills/pencil-ui-designer/`

### 4.3 实现阶段

AI工具应用：
1. 代码生成
2. 注释生成
3. 测试生成与执行
4. 多层次多级别测试（单元、集成、系统）

skill项目案例：
- `.claude/skills/vue-best-practices/`
- `.claude/skills/test-driven-development/`

### 4.4 贯穿全局

AI工具应用：
1. 项目知识规范化陈述（CLAUDE.md）
2. 上下文渐进式披露（harness engineering）
3. 设计与实现的评价评审
4. 安全审计
5. 上下文工程标准化实践
6. 版本控制与AI Coding的联动：分支治理

skill项目案例：
- `CLAUDE.md` 层级结构
- `.claude/rules/` 规范体系
- `.claude/skills/` Skills体系

## 第5章：常见问题场景的操作（20分钟）

### 5.1 大型遗留项目治理

**场景1：上下文窗口爆炸**
- 分层摘要策略：项目级→模块级→文件级
- 建立精简的架构描述文档作为"基座上下文"
- 利用目录树+关键文件组合代替全量代码

**场景2：故障定位与修复**
- 先让AI分析错误堆栈和日志，生成可能原因列表
- 人来根据业务知识筛选方向
- 再让AI在确定方向上深入排查
- 关键：不要让AI同时猜方向和找细节

**场景3：局部重构与技术债清理**
- 先让AI分析目标模块的依赖关系图
- 重构时必须同步生成回归测试
- 小步提交，每步验证

**场景4：补测试**
- 先让AI阅读代码生成行为描述
- 人确认描述是否准确
- 再基于确认后的描述生成测试用例
- 避免"给这个文件写测试"的模糊指令

**场景5：文档补全与知识抢救**
- 让AI阅读代码反向生成架构文档
- 人必须逐条审核AI输出
- 分模块进行，逐模块确认

**场景6：渐进式模块化拆分**
- 拆分策略必须人来决定（按业务域还是按技术层）
- AI只负责执行
- 让AI理解"当前状态"和"目标状态"的差异

### 5.2 新项目启动

**场景1：从需求到技术方案的"冷启动"**
- 先用AI把模糊需求结构化，产出需求文档
- 让AI提出技术方案选项并比较优劣
- 人做决策后，让AI生成详细技术设计
- 每一步产出物固化成文件，作为后续上下文输入

**场景2：项目脚手架生成**
- 让AI生成项目初始结构、配置文件、CI/CD管道
- 必须提供明确的技术栈约束和团队规范
- 建议准备标准化技术栈规范文档
- skill项目案例：Monorepo + pnpm workspace

**场景3：接口契约先行**
- 先生成完整的API契约（OpenAPI/Swagger）
- 双方基于契约独立开发
- 人需审查：字段命名一致性、错误码规范性

**场景4：原型快速验证**
- 快速生成可交互原型
- 明确告诉团队：原型代码是一次性的
- 确认方向后基于正式技术设计重新实现

**场景5：上下文工程的初始化**
- 建立CLAUDE.md + .claude/rules + Skills基础设施
- 项目规则文件：coding standards、架构约束、命名规范
- 上下文模板：每次对话的标准背景信息
- prompt库：常用任务的标准prompt

**场景6：测试策略前置**
- 基于技术设计文档生成测试策略
- 生成测试用例框架、mock数据方案
- 在写业务代码前建立测试基础设施
