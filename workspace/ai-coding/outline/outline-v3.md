- **受众**：团队内部

- **目标**：交付一套可直接复刻的「AI Coding 工具包」

## 目录结构

### Part 0: 开场（5分钟）

**01. 封面**

- 标题：AI Coding 实践分享
- 副标题：从对话到 Agent，重新定义开发流程

**02. 痛点引入**

- 你是否遇到过：
  - AI 生成的代码不符合项目规范？
  - AI 不理解你的项目架构？
  - AI 写的代码难以维护？
- 问题根源：缺乏上下文理解、没有结构化工作流、缺乏规范约束

**03. 行业趋势：从"模型崇拜"到"工程落地"**

- 行业正经历从单纯追求模型能力向工程化落地的转型
- 单纯依靠增加模型参数规模已无法解决复杂业务逻辑中的幻觉与失控问题
- 上下文管理成为 AI 编码的核心挑战
- 规格驱动开发（SDD）成为企业级落地的关键范式

**04. 方法论演进：从 Prompt 到 Harness**

- 三阶段演进：

```
Prompt Engineering → Context Engineering → Harness Engineering
     (提示词工程)          (上下文工程)          (编排工程)
```

- **Prompt Engineering**：
  - 关注"如何写更好的指令"
  - 问题：指令再好，上下文不足仍会失败
- **Context Engineering**：
  - 关注"如何为模型筛选最精准的上下文"
  - 问题：上下文管理仍依赖人工，难以复用
- **Harness Engineering**：
  - 关注"如何用工程化手段约束 AI 行为"
  - 通过模板、规则、工作流形成可复用体系
  - 这是本次分享的核心：给你一套完整的 Harness

### Part 1: 背景铺垫——AI Coding 能力边界（15分钟）

**05. AI 能力等级**

- 强：代码生成（需清晰规范）
- 需人引导：架构决策（需明确约束）
- 有风险：跨模块重构（需小步验证）
- 依赖上下文：业务逻辑理解（需上下文质量）
- 关键结论：用在什么地方决定效果

**06. 什么是规格驱动开发（SDD）**

- 一句话定义：先定义"是什么"，再定义"怎么做"
- 核心流程：
  ```
  规格定义 → 规格评审 → 规格实现 → 规格验证
  ```
- 与对话驱动的对比：
  - 对话驱动：AI 根据聊天内容猜测意图，结果不可预测
  - 规格驱动：AI 根据规格文档执行任务，结果可追溯、可验证
- 规格文档的作用：
  - 作为人机协作的"契约"
  - 先让 AI 生成规格，人评审后再执行
  - 规格可沉淀、可复用、可追溯

**07. 为什么需要"规格驱动"**

- 对话驱动的问题：
  - 意图模糊，结果不可预测
  - 需求只在聊天记录，无法沉淀
  - 缺乏组织，难以协作
  - 对话驱动的问题，让大家意识到需要把对话的内容文档化才能有效的复用（只说方案，不实现代码->Plan模式->spec驱动）
- 规格驱动的好处：
  - 可沉淀、可复用、可追溯
  - 先定义"是什么"，再定义"怎么做"

**08. 人机协作边界**

- 需求阶段：人定目标，AI 推演细节
- 设计阶段：人做决策，AI 生成方案
- 实现阶段：人定规范，AI 执行生成
- 验证阶段：人把控质量，AI 辅助测试

### Part 3: 四阶段工作流程（融合工具应用）（20分钟）

**13. 四阶段总览**

- 流程图：需求 → 设计 → 环境规范 → 实现
- 每阶段用 5W1H 定义：谁参与、做什么、标准、产出、如何做
- 关键原则：顺序不能乱，跳过环境规范阶段是最常见错误

---

#### 需求阶段

**14. 需求阶段——定义与工具应用**

- **流程**：需求推演 → 结构化描述 → PRD 文档
- **负责**：PM 主导 + SDE 参与
- **AI 可做**：
  - 需求推演完善——基于少量需求描述，AI 帮你推演补充可能遗漏的场景
  - 结构化描述——把模糊的想法转换成结构化的 Markdown 文档
  - 界面原型生成——快速生成可交互原型，加速需求确认
  - 需求文档生成——根据结构化描述自动生成 PRD 文档
- **人必须做**：业务目标定义、验收标准确定
- **产出**：PRD 文档
- **进入条件**：需求评审通过
- **配合工具**：Superpowers brainstorm
- **模板文件**：docs/requirements/prd-template.md

---

#### 设计阶段

**15. 设计阶段——定义与工具应用**

- **流程**：propose → explore → apply → archive
- **负责**：SDE 主导 + PM 参与
- **AI 可做**：
  - 架构设计与项目结构治理
  - 技术选型（对比方案优劣）
  - 接口契约设计
  - 任务分解
  - 测试场景与测试方案制定
  - 运维与持续交付方案设计
- **人必须做**：技术选型决策、架构边界定义
- **产出**：proposal + design + tasks + specs
- **进入条件**：技术评审通过
- **配合工具**：OpenSpec propose/explore
- **模板文件**：openspec/changes/ 目录结构

---

#### 环境规范阶段

**16. 环境规范阶段**

- **流程**：配置 CLAUDE.md → 配置 rules → 验证上下文
- **负责**：SDE
- **AI 可做**：辅助配置生成
- **人必须做**：编写项目规范、验证上下文有效性
- **产出**：CLAUDE.md + .claude/ 目录
- **进入条件**：AI 能正确理解项目上下文
- **⚠️ 最常见错误**：跳过此阶段直接进入实现

---

#### 实现阶段

**17. 实现阶段——定义与工具应用**

- **流程**：按 tasks 执行 → 生成代码 → 运行测试 → 质量验证
- **负责**：SDE
- **AI 可做**：
  - 代码生成——根据 specs 和 tasks 生成代码
  - 注释生成——自动生成函数注释、模块文档
  - 测试生成与执行
  - 多层次测试覆盖（单元、集成、系统）
- **人必须做**：验证输出质量、处理边界情况
- **产出**：代码 + 测试 + 文档
- **进入条件**：lint 通过、测试覆盖达标
- **配合工具**：OpenSpec apply + Superpowers TDD
- **⚠️ 重要提醒**：AI 生成的代码必须人工验证其"业务正确性"——能跑通 ≠ 有业务意义

---

#### 贯穿全局

**18. 贯穿全局——工具应用**

- 知识规范化陈述：合理组织项目知识，让 AI 能理解和复用
- 上下文渐进式披露：制定规范（spec），控制上下文披露范围
- 设计与实现评审：AI 辅助评审，发现潜在问题
- 安全审计：检查代码中的安全漏洞和风险点
- 版本控制联动：分支治理策略调整
- 上下文工程标准化：形成可复用资产

---

#### 阶段总结

**19. 阶段总结**

- 四阶段核心规律：
  | 阶段 | 核心规律 |
  |:---:|:---|
  | 需求 | 人定目标，AI 推演 |
  | 设计 | 人做决策，AI 生成方案 |
  | 环境规范 | 人定规范，AI 执行生成 |
  | 实现 | 人把控质量，AI 辅助测试 |

- 关键提醒：
  1. 顺序不能乱——跳过环境规范阶段是最常见错误
  2. 每个阶段都有进入条件——不满足就不能进入下一阶段
  3. 决策在人，执行在AI——这个边界不能模糊

### Part 4: 对现有工作的影响（10分钟）

**20. 对 PM 工作的影响**

- 需求陈述变化：需要结构化模板（PRD模板），让AI能理解需求
- 设计工作可代劳：界面原型、数据流图可交给AI生成
- 知识文档快速生成：格式规范、内容范围更标准化
- 需求变更流程调整：必须同步更新 spec 文件，AI只认文档

**21. 对 SDE/QA 的影响**

- 架构设计前移：design.md 在代码之前产出
- SDE 补测试技能：AI可生成测试用例，需验证其业务意义
- QA 参与开发：基于 tasks.md 任务分配，任务已拆得很细
- 需要学习的能力：
  - 测试思维：什么是有效测试
  - 任务拆分：如何写出 tasks.md
  - 设计文档：design.md 写什么

**22. 对评审的影响**

- 需求评审：验证AI推演结果的准确性，AI补充的细节是否符合业务预期
- Code Review：增加AI输出质量校验环节，是否符合团队规范？是否有安全隐患？
- 测试评审：验证测试用例的业务意义，能跑通 ≠ 有业务意义
- ⚠️ AI生成的代码、测试、文档都需要人工确认其"业务正确性"

**23. 人的工作调整**

- 从**执行者**到**把关者**：建立规则、审核输出
- 从**单环节**到**全流程**：了解PM/SDE/QA各环节要求
- 从**自己写**到**与AI共生**：既指导也采纳建议
- **最重要的能力**：建立对AI输出的校准直觉，能识别出"AI正在谄媚"的场景

### Part 5: 常见场景指引（5分钟）

> 说明：本部分仅介绍场景策略要点，具体实操可参考课后资料

**24. 大型遗留项目治理（要点指引）**

**场景1：上下文窗口爆炸**

- 策略：分层摘要（项目→模块→文件）
- 关键：目录树+关键文件代替全量代码

**场景2：故障定位与修复**

- 流程：AI分析堆栈→人筛选方向→AI深入
- 关键：不让AI同时猜方向和找细节

**场景3：局部重构与技术债清理**

- 流程：AI分析依赖→生成回归测试→小步验证
- 关键：每一步人工确认

**场景4：补测试**

- 流程：AI生成行为描述→人确认→生成测试
- 避坑：不要说"给这个文件写测试"

**场景5：文档补全与知识抢救**

- 流程：AI反向生成→人逐条审核
- 关键：分模块进行，逐模块确认

**场景6：渐进式模块化拆分**

- 原则：拆分策略人来定，AI只执行
- 关键：明确当前状态和目标状态

**25. 新项目启动（要点指引）**

**场景1：从需求到技术方案的"冷启动"**

- 流程：AI结构化需求→AI提方案→人决策→AI生成设计
- 关键：每一步产出物固化成文件

**场景2：项目脚手架生成**

- 内容：目录结构、配置文件、CI/CD
- 要求：必须提供明确技术栈约束

**场景3：接口契约先行**

- 流程：生成完整API契约→前后端并行
- 审查：字段命名、错误码规范性

**场景4：原型快速验证**

- 目的：快速生成可交互原型确认方向
- 提醒：原型代码是一次性的

**场景5：上下文工程的初始化**

- 内容：CLAUDE.md + rules + Skills
- 收益：前期投入持续产生回报

**场景6：测试策略前置**

- 时机：写第一行业务代码前
- 内容：测试策略、用例框架、mock数据

### Part 7: 总结（7分钟）

**28. 总结**

- **AI 不是万能的**：擅长代码生成，不擅长业务理解。决策在人，执行在AI
- **工作流要标准化**：需求、设计、实现三阶段，每个阶段有明确的进入条件和产出物
- **角色在转变**：从执行者到把关者，需要建立对 AI 输出的校准直觉
- **场景要灵活应对**：遗留项目和新项目有不同的策略，核心是处理好上下文

**29. 课后自学指引**

- 初级：会用工具，理解四阶段工作流，能用 AI 完成单个任务
- 中级：能判断质量，能判断 AI 输出是否符合业务预期，能写出合格的 spec 文档
- 高级：能设计策略，能设计上下文策略，能开发团队专属 Skill，沉淀最佳实践
