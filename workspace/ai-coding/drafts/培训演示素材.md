## 一、工作流指南

### 文件：`docs/guides/openspec-superpowers-workflow.md`

```
标准四阶段工作流：

阶段 1：探索需求 `/opsx:explore`
- 用来想清楚"做什么"，不写代码
- 输出：澄清后的需求理解

阶段 2：创建变更提案 `/opsx:propose`
- 一步生成完整的变更三件套：
  - proposal.md → 变更背景、目标、范围
  - design.md   → 技术方案、架构决策
  - tasks.md    → 细粒度实现任务列表

阶段 3：实现 `/opsx:apply`
- 按 tasks.md 逐条实现
- Superpowers Skills 在此阶段协同

阶段 4：归档沉淀 `/opsx:archive`
- 将变更记录存入项目知识库

快速参考：
| 场景 | 命令 |
|------|------|
| 需求不清楚，先聊聊 | `/opsx:explore` |
| 需求清楚，生成方案和任务 | `/opsx:propose` |
| 开始写代码 | `/opsx:apply` |
| 功能做完了 | `/opsx:archive` |
```

## 二、项目规则入口

### 文件：`CLAUDE.md`

```markdown
# 职业素养测评项目规则

## 通用约定

- 请始终保持中文对话。
- 采用Monorepo项目架构，使用 pnpm 作为包管理工具。
- ESLint + Prettier 进行代码格式化和静态检查。
- 当我更新requirements中的内容时，需要同步更新架构相关的内容。
- 应该目录下所有的CLAUDE.md文件，并该目录下内容协作的规则。
- 调用opsx:apply命令时，需要调用superpowers相关的命令来实现功能。

## 禁止事项

- 禁止修改 `.env` 文件
- 禁止直接操作数据库
- 禁止读取temp中的内容，除非明确要求
- 禁止使用 `@CreateDateColumn`、`@UpdateDateColumn`、`@DeleteDateColumn` 装饰器

## 数据库时间字段规范

所有时间字段统一使用 BIGINT 毫秒时间戳：
- Entity 继承 `BaseEntity`（普通实体）或 `SoftDeletableEntity`（软删除实体）
- 自定义时间字段使用 `type: 'bigint'`

## UI 组件开发前置检查清单（强制）

实现任何 UI 功能前，必须按顺序执行以下检查：
1. 识别 Pencil 组件
2. 查阅组件索引
3. 选择实现策略
4. 禁止行为（会被 ESLint 拦截）
```

## 五、Skills 示例

### 5.1 vue-best-practices

```yaml
name: vue-best-practices
description: MUST be used for Vue.js tasks. Strongly recommends Composition API with `<script setup>` and TypeScript.
```

**核心原则**：
- Keep state predictable: one source of truth
- Make data flow explicit: Props down, Events up
- Favor small, focused components
- Avoid unnecessary re-renders

**工作流**：
1. 确认架构后再编码（必须）
2. 应用 Vue 核心基础（必须）
3. 仅在需求需要时考虑可选特性
4. 行为正确后再进行性能优化
5. 完成前进行自查

### 5.2 test-driven-development

```yaml
name: test-driven-development
description: Use when implementing any feature or bugfix, before writing implementation code
```

**核心原则**：
```
NO PRODUCTION CODE WITHOUT A FAILING TEST FIRST
```

**Red-Green-Refactor 循环**：
1. RED：编写失败的测试
2. 验证 RED：确认测试失败原因正确
3. GREEN：编写最小代码使测试通过
4. 验证 GREEN：确认测试通过，所有测试通过
5. REFACTOR：清理代码，保持测试通过

## 六、规范文件示例

### 文件：`rules/multi-tenant-development.md`

```markdown
# 多租户开发规范

## 规则一：Entity 必须显式实现 TenantAware 接口
- 新增参与租户数据隔离的 Entity，必须 implements TenantAware
- 禁止绕过 TenantAwareRepository 直接使用 Repository<Entity>

## 规则二：先查后创建模式必须同时满足 DB 唯一约束 + 捕获 ER_DUP_ENTRY
- 数据库层唯一约束（防止并发写入脏数据）
- 代码层捕获 QueryFailedError 且 errno === 1062

## 规则三：审计日志、通知、埋点等 side effect 禁止 await
- 必须 fire-and-forget + .catch(warn) 模式

## 规则四：使用 createQueryBuilder 时自动租户过滤
- 必须使用 this.createQueryBuilder()
- 禁止使用 this.repo.createQueryBuilder()
```
