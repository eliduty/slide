# Superpowers 完全指南：给 AI Agent 装上纪律

## 一、Superpowers 是什么？

### 一句话定义

> Superpowers 是 Claude Code 的官方插件，由一组可组合的"技能"构成，让 AI 按照专业流程工作，而不是随意编写代码。

### 核心理念

```
传统 AI 编程：AI 直接写代码，结果不可预测
Superpowers：AI 先规划、再测试、后实现，结果可预测
```

### 作者背景

- **作者**：Jesse Vincent
- **身份**：Claude Code 核心贡献者，30 年软件开发经验
- **发布时间**：2025 年 10 月
- **Star 数**：106,000+（截至 2026 年 3 月）

### 为什么火得这么快？

1. **官方背书**：不是野生插件，是官方级别的产品
2. **解决真实痛点**：AI 直接写代码不问细节、代码没测试、遇到 bug 乱改
3. **开源免费**：MIT 协议，社区活跃

## 二、Superpowers 解决什么问题？

### AI 编程的三大痛点

| 痛点 | 表现 | Superpowers 的解法 |
|------|------|-------------------|
| **需求理解偏差** | AI 直接写代码，不问清楚需求 | Brainstorm 技能：先提问，再设计 |
| **代码质量不可控** | 没有测试，质量靠运气 | TDD 技能：先写测试，再写代码 |
| **执行偏离计划** | 写到一半开始乱改之前的代码 | SubAgent 技能：每个任务独立上下文 |

### 一个真实的案例

**没有 Superpowers**：
```
用户：帮我重构这个 API 模块
AI：[直接开始改文件]
AI：[改了 10 个文件后]
AI：[忘了之前的设计，开始乱改]
用户：[手动回滚 git 重来]
```

**有 Superpowers**：
```
用户：帮我重构这个 API 模块
AI：[Brainstorm] 先问清楚需求
AI：[Write Plan] 出一份详细计划
AI：[SubAgent] 每个任务独立执行
AI：[TDD] 先写测试再写代码
AI：[Review] 两轮审查
用户：[验收通过]
```

### 技能详解

#### 1. brainstorm（头脑风暴）

**触发时机**：用户提出新需求时

**核心能力**：
- 通过苏格拉底式提问精炼需求
- 探索替代方案
- 分段呈现设计方案供验证

**工作流程**：
```
用户：帮我实现用户登录功能

AI [brainstorm]：
1. 需要支持哪些登录方式？（账号密码/手机验证码/第三方登录）
2. 用户表结构是什么样的？
3. 有没有现成的认证中间件要保留？
4. 错误处理是返回 HTTP 状态码还是抛异常？
5. 需要记住登录状态吗？有效期多久？

[生成设计方案]
[分段呈现，等待确认]
```

**价值**：避免 AI 理解偏差，确保方向正确。

---

#### 2. write-plan（编写计划）

**触发时机**：设计方案批准后

**核心能力**：
- 将工作分解为 2-5 分钟的小任务
- 每个任务包含：文件路径、代码示例、验证步骤
- 计划清晰到"热情但没判断力的初级工程师也能照做"

**输出示例**：
```markdown
## 任务 1：创建 auth.repository.ts

- 文件：src/repositories/auth.repository.ts
- 从 auth.ts 提取所有数据库调用
- 导出函数：findUserByEmail, createUser, updateToken
- 验证：运行 vitest src/repositories/auth.repository.test.ts

## 任务 2：创建 auth.repository.test.ts

- 文件：src/repositories/auth.repository.test.ts
- 测试 findUserByEmail 正常和异常情况
- 测试 createUser 唯一约束冲突
- 验证：运行测试，全部通过

## 任务 3：创建 auth.service.ts

- 文件：src/services/auth.service.ts
- 依赖 auth.repository.ts
- 包含：login, register, refreshToken, logout
- 验证：运行测试
```

**价值**：让 AI 执行有章可循，不会偏离。

---

#### 3. execute-plan（执行计划）

**触发时机**：计划批准后

**核心能力**：
- 按顺序执行每个任务
- 每个任务完成后验证
- 失败时自动回滚

---

#### 4. test-driven-development（测试驱动开发）

**触发时机**：实现代码时

**核心能力**：
- 强制执行 RED-GREEN-REFACTOR 循环
- **如果发现代码写在测试前面，框架会删掉那段代码**

**工作流程**：
```
1. RED：先写测试，运行测试，确认失败
2. GREEN：写最少代码让测试通过
3. REFACTOR：重构代码，保持测试通过
```

**价值**：确保代码质量，不是"写完再补测试"。

---

#### 5. systematic-debugging（系统化调试）

**触发时机**：遇到 bug 时

**核心能力**：4 阶段根本原因分析

```
阶段 1：复现问题
- 确认问题的具体表现
- 找到最小复现路径

阶段 2：定位根因
- 分析错误堆栈
- 追踪数据流
- 识别问题代码

阶段 3：实施修复
- 编写失败的测试用例
- 修复代码
- 确认测试通过

阶段 4：防御性编程
- 添加边界检查
- 改进错误处理
- 更新文档
```

**价值**：不是"遇到 bug 就乱改"，而是系统化解决。

---

#### 6. subagent-driven-development（子代理驱动开发）

**触发时机**：执行计划时

**核心能力**：
- 每个任务分配一个独立的子 Agent
- 子 Agent 拿到全新的上下文，不会"记忆混乱"
- 多个任务可以并行执行

**为什么需要子 Agent？**

```
单 Agent 问题：
[任务 1] → [任务 2] → [任务 3] → ...
   ↓           ↓           ↓
上下文越来越乱，开始偏离计划

多 Agent 方案：
[任务 1] → Agent A（独立上下文）
[任务 2] → Agent B（独立上下文）
[任务 3] → Agent C（独立上下文）
```

**两阶段审查**：
1. **规范符合性审查**：是否符合需求规格？
2. **代码质量审查**：代码风格、最佳实践

---

#### 7. use-git-worktrees（使用 Git Worktrees）

**触发时机**：开始新功能时

**核心能力**：
- 在新分支上创建隔离的工作空间
- 不污染 main 分支
- 完成后可以选择合并、发 PR、或丢弃

---

#### 8. request-code-review（请求代码审查）

**触发时机**：任务完成后

**核心能力**：
- 预审查检查清单
- 确保代码符合团队标准

### 技能触发规则

| 技能 | 触发条件 | 是否强制 |
|------|----------|---------|
| brainstorm | 用户提出新需求 | 强制 |
| write-plan | 设计批准后 | 强制 |
| execute-plan | 计划批准后 | 强制 |
| TDD | 实现代码时 | 强制 |
| systematic-debugging | 遇到 bug 时 | 强制 |
| subagent | 执行任务时 | 自动 |
| git-worktrees | 开始新功能时 | 自动 |

**重要**：Superpowers 的技能不是"建议"，而是"强制"。当技能适用时，AI 必须使用它。

## 六、实战案例

### 案例：重构 API 模块

**需求**：重构 src/auth.js，把路由、业务逻辑、数据库操作拆分到独立模块，加上单元测试。

**没有 Superpowers**：
```
AI：[直接开始改文件]
AI：[改了 10 个文件]
AI：[忘了之前的设计]
用户：[手动回滚]
```

**有 Superpowers**：

```
用户：帮我重构 src/auth.js，把路由、业务逻辑、数据库操作拆分到独立模块

AI [brainstorm]：
1. 用什么测试框架？
用户：Vitest

2. 数据库用的是什么 ORM？
用户：Prisma

3. 有没有现成的认证中间件要保留？
用户：有，JWT 验证

4. 错误处理是返回 HTTP 状态码还是抛异常？
用户：状态码

[生成设计方案]
拆成四个文件：auth.routes.js、auth.service.js、auth.repository.js、auth.middleware.js

用户：确认

AI [write-plan]：
生成 8 个任务...

用户：确认

AI [execute-plan]：
任务 1：创建 auth.repository.js
  [SubAgent] 先写测试 → 测试失败 → 写代码 → 测试通过 → Review
任务 2：创建 auth.repository.test.ts
  [SubAgent] ...
任务 3：创建 auth.service.js
  [SubAgent] ...
...

AI [收尾]：
所有任务完成，全量测试通过。
请选择：合并 / 发 PR / 保留 / 丢弃
```

**结果**：
- 8 个任务全部完成
- 每个任务都有测试
- 代码经过两轮审查
- AI 全程独立执行，人工干预 < 5 条指令

## 七、Superpowers 的核心价值

### 1. 让 AI 编程可预测

```
传统 AI 编程：输入 → [黑盒] → 输出（不确定）
Superpowers：输入 → [结构化流程] → 输出（可预测）
```

### 2. 强制执行最佳实践

- 先设计再编码
- 先测试再实现
- 每个任务独立上下文
- 两轮审查

### 3. 降低协作成本

- 计划文档可共享
- 团队成员可以审查 AI 的计划
- 规范一致，代码风格统一

### 4. 提升代码质量

- TDD 不是可选项
- 系统化调试
- 防御性编程

## 八、与其他工具的关系

### Superpowers vs Skills

| 概念 | 说明 |
|------|------|
| **Skills** | Claude Code 的通用技能系统，用户可自定义 |
| **Superpowers** | 基于 Skills 构建的完整工作流，官方出品 |

Superpowers 是 Skills 的"最佳实践集合"。

### Superpowers vs OpenSpec

| 工具 | 定位 | 关系 |
|------|------|------|
| **OpenSpec** | 规范框架，生成项目文档 | 可独立使用 |
| **Superpowers** | 执行框架，确保 AI 按流程工作 | 可独立使用 |
| **组合使用** | OpenSpec 生成规范 → Superpowers 执行 | 更强大 |

## 九、总结

### Superpowers 的本质

> Superpowers 不是让 AI 更聪明，而是让 AI 更有纪律。

### 核心价值

1. **可预测性**：结构化流程带来可预测的结果
2. **质量保证**：TDD、系统化调试、两轮审查
3. **协作友好**：计划文档可共享，团队可审查
4. **降低风险**：Git Worktrees 隔离，失败可回滚

### 适用场景

- ✅ 新功能开发
- ✅ 复杂重构
- ✅ Bug 修复
- ✅ 团队协作项目

### 不适用场景

- ❌ 简单修改（流程开销大于收益）
- ❌ 探索性原型（需要快速迭代）

### 一句话总结

**Superpowers 让 AI 从"热情但没纪律的实习生"变成"严格遵循 SOP 的专业工程师"。**
